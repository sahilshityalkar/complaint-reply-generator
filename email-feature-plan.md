# ReplyAI — Send Reply via Email: Complete Feature Plan

> **PM's note:** This is not just a "send email" button. This is the feature that turns ReplyAI from a copy-paste tool into a real business tool. Get this right and users never leave the app.

---

## The Core Decision: Whose Email?

Every "send as" feature has one fundamental question: **whose email address does the customer see?**

There are three options. Here's the honest breakdown:

### Option A: ReplyAI's Domain (ReplyAI sends on your behalf)

```
From: "Sarah from Handmade with Heart" <replies@replyai.app>
```

| Pros | Cons |
|------|------|
| Zero setup for user — works immediately | Customer sees "replyai.app" not the business domain |
| We control deliverability (SPF, DKIM) | Looks slightly less professional |
| No DNS, no OAuth, no SMTP config | Can't reply back to the thread |

**Verdict:** Best for launch. Ship this first.

### Option B: User's Own Email Domain (User verifies their domain)

```
From: "Sarah from Handmade with Heart" <sarah@handmadewithheart.com>
```

| Pros | Cons |
|------|------|
| Fully branded — customer sees the real business | User must add DNS records (SPF, DKIM, MX) |
| Professional, trustworthy | Support nightmare — "my DNS isn't working" |
| Builds long-term trust | Most micro-business owners won't know how |

**Verdict:** Premium feature for Pro plan. Build later.

### Option C: Gmail OAuth / SMTP (User connects their inbox)

```
From: "Sarah" <sarah@gmail.com>
```

| Pros | Cons |
|------|------|
| Simple for the user ("Login with Google") | Most businesses don't use @gmail.com |
| No DNS required | Gmail rate limits (500/day) |
| | OAuth token management is fragile |

**Verdict:** Too much maintenance overhead. Skip.

---

## The Decision: Phase 1 = Option A, Phase 2 = Option B

```
Launch:      ReplyAI domain → dead simple, zero setup, works instantly
Pro plan:    Custom domain → user adds DNS, fully branded
```

```

                                                                              ↑
```

## Architecture

### Database Changes

Add to `reply_history`:
```sql
alter table reply_history add column sent_via_email boolean default false;
alter table reply_history add column customer_email text;
alter table reply_history add column email_subject text;
alter table reply_history add column sent_at timestamptz;
```

### New API Endpoint

```
POST /api/email/send

Headers: Authorization (Clerk session)
Body: {
  reply_text: "I completely understand...",
  customer_email: "angrycustomer@gmail.com",
  subject: "Re: Your recent order — we're on it",
  profile_id: "uuid-of-brand-profile",   // optional
  business_name: "Handmade with Heart",  // for from name
  reply_id: "uuid"                        // which reply was this
}

Response: {
  success: true,
  email_id: "resend-email-id"
}
```

### Resend Integration

The email sending flow:

```typescript
import { Resend } from 'resend';

const resend = new Resend(process.env.RESEND_API_KEY);

export async function sendReply(params: {
  to: string;
  subject: string;
  reply_text: string;
  business_name: string;
  sign_off?: string;
}) {
  const { to, subject, reply_text, business_name, sign_off } = params;

  const fromName = `${business_name} via ReplyAI`;
  const body = sign_off
    ? `${reply_text}\n\n${sign_off}`
    : reply_text;

  const { data, error } = await resend.emails.send({
    from: `ReplyAI <replies@replyai.app>`,     // verified domain
    to: [to],
    subject,
    text: body,
    html: body.replace(/\n/g, '<br>'),
  });

  if (error) throw error;
  return data;
}
```

### From Address Strategy

**Launch approach — single verified domain:**

1. Verify `replyai.app` (or whatever domain you use) in Resend
2. Add these DNS records: SPF, DKIM
3. All emails sent from `replies@replyai.app`
4. "From name" is dynamic: `"Handmade with Heart via ReplyAI"`

**Why "via ReplyAI":**
- Trust — customer knows it came through a tool, not spoofed
- Transparency — reduces spam reports
- Marketing — every email is a subtle ad for ReplyAI

### Plan Limits

| Plan | Email Sends/Month |
|------|-------------------|
| Free | 5 sends |
| Starter ($9) | 50 sends |
| Pro ($19) | Unlimited |

**Why 5 on free:** Enough to experience the magic. They'll hit the limit fast and want more.

### Rate Limiting & Abuse Prevention

- Max 1 email per 5 seconds per user (prevent spam)
- If Resend returns an error, we retry once (3-second delay)
- If customer email bounces, we log it but don't show the user (protect Resend reputation)
- We track `sent_at` to enforce per-plan limits

## Error Handling

| Scenario | What Happens |
|----------|-------------|
| Resend API down | Show "Email service unavailable. Reply copied to clipboard — paste it manually." |
| Invalid email format | Client-side validation before sending |
| Plan limit reached | Show UpgradeModal (already built) |
| Rate limit hit | Toast: "Sending too fast. Please wait a few seconds." |
| Resend bounce | Log silently, don't disturb user |
| User not authenticated | 401 (Clerk middleware handles) |
| Missing reply_text | 400 validation error |
